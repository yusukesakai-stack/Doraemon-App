# 技術定義書_Vibe Walker

## 1. ねらい
このドキュメントは、エンジニアでない方にも分かるように「Vibe Walkerが何でできていて、どうやって動くか」をやさしく説明します。

## 2. 全体構成図（テキスト）
```
[ユーザーのスマホ/PCブラウザ]
        │ (HTTPS)
        ▼
[Cloud Run 上のWebアプリ]──┐
        │                   │
        ▼                   ▼
[Google Maps API]    [Gemini API (AI)]
        │                   │
        └────► ルート＋見どころを生成 ◄────┘
```
- **ブラウザ**: ユーザーが触る画面。スマホでも軽く動く。
- **Cloud Run**: Googleのクラウド上でアプリを動かす場所。アクセスがあるときだけ動くので安い。
- **Google Maps API**: 地図と道案内のデータ提供役。
- **Gemini API**: 賢いAI。ルートの「気持ちよさ」を判断したり、見どころの紹介文を書いたりする。

## 3. 各パーツの役割
| パーツ | 役割 | やさしい説明 |
|---|---|---|
| フロントエンド (React + Vite) | 画面表示 | ボタンや地図を表示する「お店の看板と内装」 |
| バックエンド (Node.js / Hono) | 司令塔 | ブラウザからの注文を受けてAPIに問い合わせる「店員さん」 |
| Google Maps Directions API | 道順計算 | カーナビのような役割 |
| Google Maps Places API | 場所情報 | お店や公園の情報を持っている図鑑 |
| Gemini API | AI判定 | ルートの良さを評価し、紹介文を書く頭脳 |
| Cloud Run | 実行環境 | アプリを置いておく「貸しスペース」 |
| GitHub | 保管庫 | アプリの設計図をバックアップ |
| Cloud Build | 自動組立 | GitHub更新→自動でCloud Runへ反映 |

## 4. データの流れ
1. ユーザーが「30分」をタップ
2. ブラウザが現在地をGPSで取得
3. バックエンドへ「30分・現在地」を送信
4. バックエンドがDirections APIに複数の周回ルート候補を問い合わせ
5. 各ルートについてPlaces APIで沿道スポットを取得
6. Gemini APIが「緑が多い／静か／見どころあり」で評価＆紹介文作成
7. 最良ルートと紹介文をブラウザに返す
8. 地図上にルート＆ピンを表示

## 5. インフラ構成
- **ホスティング**: Google Cloud Run（コンテナ）
- **ビルド**: Cloud Build + GitHub Actions
- **シークレット管理**: Secret Manager（APIキー保護）
- **モニタリング**: Cloud Logging / Cloud Monitoring

## 6. セキュリティ
- HTTPS必須
- APIキーはサーバー側で保持（ブラウザに出さない）
- 位置情報はサーバーに保存しない（処理のみ）
- レート制限で乱用防止

## 8. 専門用語のやさしい補足
- **API**: アプリ同士が会話するための「窓口」
- **コンテナ**: アプリ一式を箱詰めしたもの。どこでも同じ動きをする
- **CI/CD**: コードを更新したら自動でテスト→公開する仕組み
